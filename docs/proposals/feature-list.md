## 说明

本文档汇总 AI 沙箱平台的完整特性清单，按功能域分组，标注各特性在 e2b 和 kuasar-sandbox 中的支持状态。旨在作为能力对比和路线图规划的基准。

状态标注：
- ✅ 已支持（完整实现）
- 🔶 部分支持（有限制或缺关键细节）
- ❌ 不支持
- 📋 规格已设计但未实现

## 十五、用户故事（User Stories）

Story 格式：**角色 / 需求 / 价值 / 关联特性 / 验收标准**。  
角色缩写：`开发者` = AI 应用开发者，`Agent` = LLM 编排层 / AI Agent，`运营者` = 平台运营工程师，`SRE` = 平台可靠性工程师，`合规官` = 企业合规 / 安全审计员，`企业管理员` = 企业 IT / 租户管理员。

### 一、沙箱生命周期管理

#### S-01：即时创建沙箱

**角色**：开发者  
**需求**：我希望通过模板 ID 创建沙箱，并指定 CPU 数量、内存大小和环境变量，几秒内拿到可用的沙箱。  
**价值**：LLM 生成代码后可立即执行，不让用户感知底层启动延迟。  
**关联特性**：从模板创建沙箱 · 快照恢复启动（snp 模板）· 创建时指定 CPU/内存 · 注入环境变量 · 指定用户自定义 metadata  
**验收标准**：
- snp 模板创建后 P50 就绪延迟 < 3 s
- 环境变量在沙箱内 `printenv` 可见
- CPU/内存限制在 cgroup v2 中正确设置

---

#### S-02：Agent 会话暂停与恢复

**角色**：Agent  
**需求**：我希望将正在运行的沙箱暂停为全内存快照，在之后任意时刻恢复，恢复后进程树和内存状态与暂停前完全一致。  
**价值**：Agent 空闲时不消耗计算资源，唤醒后继续上次任务，无需重新初始化上下文。  
**关联特性**：暂停（全内存快照）· Resume（内存快照恢复）· 快照前内存回收 · 快照前文件系统冻结 · 快照前 heap 收拢  
**验收标准**：
- 恢复后进程树与快照前完全一致（PID、打开文件、网络连接）
- P50 恢复延迟 < 80 ms（snp 模板）
- 暂停期间不计费

---

#### S-03：透明自动恢复（用户无感知）

**角色**：开发者

- 旧 keytag 的数据仍可解密
- 集群节点间密钥通过 mTLS 通道分发

---

#### S-27：严格多租户隔离

**角色**：租户管理员 / 运营者  
**需求**：我希望租户 A 访问租户 B 的资源时返回 404，管理员能随时封禁违规租户（立即终止其所有沙箱），同时支持企业 OAuth2/SSO 登录。  
**价值**：租户间信息零泄露，违规处置有手段，企业接入无需管理额外凭证。  
**关联特性**：多租户数据隔离（跨租户返回 404）· 租户封禁（team blocking）· OAuth2 认证 · Secure 模式（envd 双重鉴权）  
**验收标准**：
- 跨租户访问返回 404，不暴露资源存在
- 封禁操作 10 s 内终止租户所有沙箱
- OAuth2 OIDC flow 通过企业 IdP 完成认证

### 九、可观测性

#### S-28：沙箱运行时指标 API

**角色**：开发者 / SRE  
**需求**：我希望通过 API 查询任意沙箱的实时 CPU、内存、磁盘用量，并能从 Prometheus 端点拉取完整平台指标集。  
**价值**：开发者可定位资源异常 Agent，SRE 可设置告警，平台 SLO 可量化测量。  
**关联特性**：沙箱运行时指标（CPU/内存/磁盘）· 沙箱启动延迟指标 · 节点级资源指标 · Prometheus 完整指标清单 · SLO/SLA 定义  
**验收标准**：
- `GET /sandboxes/{id}/metrics` 返回当前 CPU/内存/磁盘
- Prometheus `/metrics` 包含 create 延迟直方图、admission 拒绝率、快照恢复耗时
- SLO：P99 create 延迟 < 3 s，P50 resume 延迟 < 100 ms

#### S-29：沙箱日志 API（v1/v2）

**角色**：开发者 / SRE  
**需求**：我希望通过 API 分页读取沙箱内应用日志，支持方向（正向/反向）、cursor、日志级别和全文搜索过滤。  
**价值**：无需 SSH 到节点即可调试 Agent 行为，日志可接入团队现有的 Loki/ELK 可观测栈。  
**关联特性**：沙箱日志 API（v1）· 沙箱日志 API（v2：方向/cursor/level/搜索）· 结构化日志聚合（Loki/ELK 兼容）· ClickHouse 指标存储  
**验收标准**：
- `GET /sandboxes/{id}/logs?level=error&cursor=xxx` 返回分页日志
- 全文搜索在 1 s 内返回结果
- 日志可通过 syslog forward 接入 Loki/ELK

---

#### S-30：分布式链路追踪

**角色**：SRE  
**需求**：我希望 create 请求从 API 网关到快照恢复到 envd 就绪的完整调用链都有 OpenTelemetry span，可在 Jaeger/Tempo 中查看。  
**价值**：定位延迟问题时不需要逐节点 SSH，秒级定位是网络、磁盘、快照还是 envd 初始化引入了延迟。  
**关联特性**：OpenTelemetry 分布式追踪  
**验收标准**：
- trace 包含 api → orchestrator → node-ctl → envd 各阶段 span
- 每个 span 有耗时和状态码
- 异常 span 可触发告警

#### S-31：持久审计日志（合规导出）

**角色**：合规官  
**需求**：我希望所有控制面操作（create/pause/kill/admin）都写入结构化审计日志，日志持久化且不随节点重启丢失，可导出到 SIEM 系统。  
**价值**：满足 SOC2、ISO27001 和金融监管对操作审计记录的强制要求，合规审查有据可查。  
**关联特性**：审计日志（控制面操作）· 审计日志持久化 · 结构化日志聚合（Loki/ELK 兼容）  
**验收标准**：
- 每条日志包含：operator_id、timestamp、action、sandbox_id、result
- 节点重启后历史日志完整保留
- 支持 syslog forward 到 Splunk/Elasticsearch

### 十、资源管控

#### S-32：per-tenant 配额与 API 限流

**角色**：运营者  
**需求**：我希望为每个租户设置最大沙箱数、并发构建数和 API 请求速率上限，超额时明确拒绝并返回有意义的错误。  
**价值**：单个失控租户无法影响其他租户的服务质量，平台可实施分级服务套餐（Free/Pro/Enterprise）。  
**关联特性**：per-tenant quota（最大沙箱数）· per-tenant 并发构建数限制 · per-tenant API 请求限流  
**验收标准**：
- 沙箱数超配额时 create 返回 429 并说明原因
- API 请求超速时返回 429 + `Retry-After` 头
- 并发构建超限时新构建进入等待队列或被拒绝

---

#### S-33：多维度资源隔离（cgroup v2）

**角色**：运营者  
**需求**：我希望每个沙箱受 cgroup v2 约束：CPU 时间片、磁盘 I/O 速率、网络发送速率、熵源访问速率均有上限。  
**价值**："噪声邻居"沙箱不能耗尽节点资源，同节点其他沙箱性能不受影响。  
**关联特性**：cgroup v2 资源隔离 · 磁盘 I/O 速率限制 · 网络发送速率限制 · 熵源设备速率限制  
**验收标准**：
- 沙箱在持续 CPU 压力下不超过声明配额
- 磁盘 I/O 和网络发送分别被限制在配置值
- `/dev/random` 读取速率受限，不影响 CSPRNG 可用性

### 十一、运营与计费

#### S-34：精确用量计量与计费事件

**角色**：运营者  
**需求**：我希望每次沙箱启停都产生计量事件，包含启动时间、停止时间、vCPU 数、内存 MB，并推送到外部计费系统，支持团队级聚合统计。  
**价值**：按秒粒度精确计费，账单可审计，客户可查看自己的用量明细。  
**关联特性**：用量计量（启停时间 × 规格）· 计费事件推送（外部系统集成）· 团队级用量聚合统计  
**验收标准**：
- 每次 create/pause/kill 产生 usage event（sid, team_id, start_at, stop_at, cpu, memory_mb）
- 事件 webhook 或消息队列推送，延迟 < 10 s
- `GET /usage?team_id=x&from=...&to=...` 返回汇总用量

---

#### S-35：Webhook 事件推送

**角色**：开发者  
**需求**：我希望订阅沙箱状态变更和构建完成等事件的 Webhook，当事件发生时平台主动推送，失败时自动重试。  
**价值**：CI/CD 流水线和 Agent 编排系统可以被动感知事件，不需要轮询 API，降低延迟和带宽消耗。  
**关联特性**：Webhook 事件推送（状态变更等）  
**验收标准**：
- 事件到达 Webhook 端点的延迟 < 5 s
- 失败重试采用指数退避，最多重试 5 次
- 支持事件类型：sandbox.created / sandbox.paused / sandbox.killed / build.completed / build.failed

### 十二、集群与多节点

#### S-36：弹性集群扩缩容与节点排空

**角色**：运营者  
**需求**：我希望新节点加入集群后自动参与沙箱调度，维护节点时能排空并将沙箱透明迁移到其他节点，整个过程用户无感知。  
**价值**：容量可弹性伸缩，节点维护不影响用户，运维操作简单可靠。  
**关联特性**：多节点集群调度 · 节点心跳与健康检测 · 节点排空（drain）· 沙箱跨节点迁移 · 节点亲和性（resume 优先原节点）· 放置调度（P2C 负载均衡）· 集群全局沙箱列表  
**验收标准**：
- 新节点 30 s 内开始接收新沙箱
- drain 5 min 内完成节点上所有沙箱迁移
- 全局沙箱列表跨节点汇聚，不遗漏

---

#### S-37：缓存亲和调度（Maglev shuffle-sharding）

**角色**：运营者  
**需求**：我希望沙箱在跨节点迁移后，调度器能将其分配到拥有对应 L2 缓存的节点，而不是随机节点。  
**价值**：迁移后恢复速度与本地一致，消除迁移引发的冷起效应。  
**关联特性**：shuffle-sharding 调度（缓存亲和）· Maglev shuffle-sharding 缓存亲和  
**验收标准**：
- 同一 content key 始终路由到固定 L2 shard 集合
- 迁移后 L2 cache hit rate 与迁移前一致
- 单节点故障只影响有界数量的 key（不影响全部）

### 十三、桌面与 Computer Use

#### S-38：虚拟桌面环境 + 鼠标键盘注入

**角色**：Agent（Computer Use）  
**需求**：我希望沙箱内有完整 Linux 桌面环境（Xvfb + 窗口管理器），并能通过 API 注入鼠标事件（点击、移动、拖拽）和键盘事件（文本输入、快捷键）。  
**价值**：Agent 可以像人一样操控 GUI 应用，实现浏览器自动化、表单填写、桌面应用测试等高价值场景。  
**关联特性**：虚拟桌面环境（Xvfb + WM）· 鼠标事件注入（xdotool）· 键盘事件注入 · 窗口管理（获取窗口列表/焦点）· 浏览器预装（Chrome/Firefox）  
**验收标准**：
- `sandbox.leftClick(x, y)` 通过 XTest 注入合成事件
- `sandbox.type("text")` 在焦点窗口中输入文本
- `getWindows()` 返回当前所有窗口及标题

#### S-39：截图 API + 桌面流传输

**角色**：Agent（Computer Use）/ 开发者  
**需求**：我希望随时截取沙箱桌面当前状态（PNG/base64），并能将桌面以 VNC over WebSocket 的方式流式传输到浏览器中实时查看。  
**价值**：Agent 获得视觉反馈做下一步决策，人工监督者可在浏览器中实时观察 Agent 的操作过程。  
**关联特性**：截图 API · 屏幕流传输（VNC over WebSocket）  
**验收标准**：
- `screenshot()` 返回当前帧 PNG，延迟 < 200 ms
- noVNC URL 在沙箱启动后 3 s 内可访问
- 流帧率 ≥ 15 fps（正常交互负载下）

### 十四、Feature Flag 体系

#### S-40：运行时特性开关（灰度发布与紧急关闭）

**角色**：平台产品工程师  
**需求**：我希望通过运行时 feature flag 按租户/团队控制特性开启状态，新特性先灰度给部分用户，出现问题时能在 30 s 内对所有用户关闭。

- flag 可定向到特定 team_id 或按百分比灰度
- 关键运营参数（连接数上限、回收超时）通过 flag 动态调整
