# AI 沙箱平台通用 Use Case 分析

**版本** v1.0 · **日期** 2026-06-23

---

## 概述

本文档描述 AI 沙箱平台的通用使用场景，作为特性规划和能力对比的基准参照。场景描述与具体产品无关，聚焦"用户在做什么、需要什么保障"。

---

## 一、场景分类框架

```
AI 沙箱平台用户场景
├── UC-A  代码执行类          AI Agent 驱动的代码运行与结果反馈
├── UC-B  数据处理类          数据集分析、转换、可视化
├── UC-C  环境构建类          构建可复用的运行时环境
├── UC-D  长时任务类          多步骤、多轮次、持久会话任务
├── UC-E  大规模并发类        批量/RL/评估等高并发场景
├── UC-F  桌面交互类          GUI 操控与视觉感知
├── UC-G  平台运营类          多租户管理、计费、运维
└── UC-H  安全合规类          隔离、审计、出站管控
```

---

## 二、详细 Use Case

### UC-A：代码执行（AI 驱动）

**背景**：LLM 生成代码后需要在真实环境中执行，获取输出或错误，再反馈给模型进行迭代。

#### UC-A1：交互式代码解释器

> 用户向 AI 提问，AI 生成 Python/JavaScript/Shell 代码，代码在沙箱中执行，输出实时返回用户。

**关键需求**：
- 毫秒级到秒级启动，不让用户等待
- stdin/stdout/stderr 流式返回
- 多轮对话中环境状态持续（变量、已安装包保留）
- 执行结果（文件、图片）可被读取和下载
- 执行超时保护，不阻塞后续对话

**成功标准**：用户感知"AI 直接帮我运行了代码并给出结果"，无感知底层沙箱的存在。

---

#### UC-A2：自主编程 Agent

> AI Agent 自主拆解任务，循环执行：生成代码 → 运行 → 观察输出 → 修正 → 再运行，直到任务完成。

**关键需求**：
- 支持 Agent 长时间驻留（数分钟到数小时）
- 会话可被暂停（Agent 睡眠）和唤醒（Agent 继续），状态保留
- Agent 可操作文件系统（读写、上传、下载）
- Agent 可安装依赖包（pip install、npm install）
- 并发安全：同一 Agent 会话内串行执行，不同 Agent 完全隔离

---

#### UC-A3：多语言运行环境

> 同一任务流程中 Agent 需要运行不同语言的代码（Python 数据处理 → Go 性能测试 → Bash 脚本验证）。

**关键需求**：
- 环境支持任意语言和运行时（Python、Node.js、Go、Java、Rust……）
- 语言间结果通过文件系统传递
- 各语言 runtime 预装，避免每次 `apt install` 的启动延迟

---

### UC-B：数据处理与分析

#### UC-B1：AI 数据分析

> 用户上传 CSV/Excel/数据库导出文件，AI 在沙箱中运行 pandas/polars/SQL，生成统计摘要和可视化图表。

**关键需求**：
- 大文件上传（100MB 级别）不阻塞
- 数据在沙箱内不离开（隐私保护）
- 生成的图表（PNG/SVG）可被下载
- 执行过程中间状态可查询（进度感知）

---

#### UC-B2：批量数据转换

> 以 Agent 编排多个沙箱并行处理数据集的不同分片，最终汇总。

**关键需求**：
- 同时运行数十到数百个沙箱，各处理一个数据分片
- 沙箱间结果通过共享存储或 API 汇聚
- 任意沙箱失败不影响其他（独立故障域）
- 全部完成后批量释放资源

---

### UC-C：环境构建与模板管理

#### UC-C1：自定义运行时构建

> 开发者在标准基础镜像上安装依赖、配置环境、预热服务，将最终状态固化为可复用模板，后续沙箱从此模板启动。

**关键需求**：
- 支持 Dockerfile 等标准容器描述方式定义环境
- 构建日志实时可见，支持调试
- 构建产物版本化管理，支持多版本并存
- 后续从模板启动的沙箱首次请求即可用（无冷启动等待）
- 构建过程与生产沙箱完全隔离（构建失败不影响已运行沙箱）

---

#### UC-C2：模板迭代与版本管理

> 团队频繁迭代运行时环境（安装新包、更新配置），需要管理多个版本，支持快速回滚。

**关键需求**：
- 模板有可读的版本标签（如 `data-env:v3`），不只是内容哈希
- 新旧版本可并存，沙箱可按版本创建
- 增量构建：仅重新构建变更层，不重跑全部步骤
- 模板元数据（创建时间、构建日志、大小）可查询

---

#### UC-C3：预热池（Warm Pool）

> 平台预先启动一批沙箱并保持就绪状态，用户请求到来时即刻分配，实现真正零延迟。

**关键需求**：
- 平台可批量从同一模板快照恢复出大量沙箱
- 预热沙箱在等待期间资源占用极低（内存按需加载）
- 用户请求分配到预热沙箱后，个性化初始化（注入 env、切换用户）耗时 < 1s
- 支持按模板分别设置池大小和预热策略

---

### UC-D：长时任务与持久会话

#### UC-D1：深度研究 Agent

> Agent 接收复杂研究任务（如"分析过去五年 GPU 市场数据"），在沙箱中持续运行数小时，访问网络、处理数据、撰写报告。

**关键需求**：
- 会话最长可持续数小时甚至数天
- 空闲时自动休眠（节省资源），有请求时自动唤醒（对 Agent 透明）
- 唤醒后完整恢复运行状态（内存、文件、进程）
- 任务进度可持久化到外部存储（断电不丢失关键中间结果）
- 支持中途将会话迁移到不同节点（节点维护场景）

---

#### UC-D2：有状态开发环境

> 开发者通过 AI 驱动的编程助手在沙箱中编写、测试代码，多次对话间环境保持不变（已安装的包、已创建的文件、已运行的服务）。

**关键需求**：
- 对话间环境状态 100% 保留（不因超时重置）
- 运行中的服务（数据库、Web 服务器）在会话恢复后继续可用
- 开发者可随时暂停会话（下班）、次日恢复
- 支持将环境状态导出为新模板（固化开发环境）

---

#### UC-D3：工作流编排（多 Agent 协作）

> 编排层将复杂任务拆分为子任务，多个 Agent 在各自沙箱中并行执行，结果汇聚后继续下一阶段。

**关键需求**：
- 父 Agent 可动态创建、监控、终止子 Agent 沙箱
- 子 Agent 间结果通过共享存储或 API 传递
- 子任务失败时父 Agent 可感知并决策（重试 / 降级 / 放弃）
- 整体任务的资源用量可被追踪和计费

---

### UC-E：大规模并发

#### UC-E1：强化学习环境（RL Training）

> RL 训练需要同时运行数万个沙箱作为独立环境，每个沙箱执行策略、计算 reward，训练循环高速迭代。

**关键需求**：
- 单次启动数千到数万个沙箱，全部从同一快照 fork（保证初始状态一致）
- 沙箱间完全隔离（一个崩溃不影响其他）
- 沙箱生命周期极短（秒级到分钟级），需要高效批量销毁
- 资源利用率最大化（低内存占用，高密度承载）
- reward 计算结果高效汇聚到训练进程

---

#### UC-E2：大规模 LLM 评估

> 在大量沙箱中并发运行 benchmark 任务，每个沙箱执行一组测试用例，汇总评估指标。

**关键需求**：
- 同时运行数百到数千个评估沙箱
- 各沙箱运行相同基准代码，但输入不同
- 任意失败任务可被重新调度（幂等执行）
- 结果数据可被批量读取（无需逐一轮询）

---

### UC-F：桌面交互与 Computer Use

#### UC-F1：GUI 自动化

> AI Agent 需要操控桌面 GUI 应用（浏览器、办公软件、IDE），在虚拟桌面中执行点击、输入、截图等操作。

**关键需求**：
- 沙箱内运行完整 Linux 桌面环境（含窗口管理器）
- Agent 可发送鼠标事件（点击、移动、拖拽）
- Agent 可发送键盘事件（输入文本、快捷键）
- 沙箱当前屏幕可被截图并返回给 Agent（视觉反馈）
- 屏幕可以流式传输到浏览器（人工监督）

---

#### UC-F2：浏览器自动化

> AI Agent 在沙箱内控制真实浏览器（Chrome/Firefox）完成网页操作（表单填写、信息抓取、Web 测试）。

**关键需求**：
- 沙箱内可运行有头浏览器（非 headless）
- 浏览器页面截图可实时获取
- 浏览器访问外网受策略控制（只允许特定域名）
- 会话 Cookie/状态在 Agent 多轮操作间保持

---

#### UC-F3：应用预览（Vibe Coding）

> AI 生成完整 Web 应用后，在沙箱中运行并暴露访问地址，用户在浏览器中实时预览。

**关键需求**：
- 沙箱内运行任意 Web 框架（Next.js、FastAPI、Streamlit……）
- 用户端口自动发现并暴露（无需声明）
- 用户可通过浏览器实时访问沙箱内的 Web 服务
- WebSocket 服务正常工作（实时更新类应用）

---

### UC-G：平台运营

#### UC-G1：多租户隔离管理

> 平台运营商为多个企业客户提供沙箱服务，需要保证资源、数据、密钥的租户间完全隔离。

**关键需求**：
- 租户 A 无法看到或访问租户 B 的沙箱
- 每个租户有独立的资源配额（最大沙箱数、并发数、存储）
- 租户超额时有明确拒绝和告警，不影响其他租户
- 租户密钥独立，平台运营商不能以租户身份操作

---

#### UC-G2：用量计量与计费

> 平台需要按资源规格和使用时长精确计量每个租户的用量，作为计费依据。

**关键需求**：
- 记录每个沙箱的：启动时间、停止时间、CPU 数、内存 MB
- 计量粒度达秒级
- 计量数据不可被篡改（审计要求）
- 支持将计量事件推送到外部计费系统

---

#### UC-G3：运维与可观测性

> 平台运维团队需要监控所有节点和沙箱的状态，及时发现和处理异常。

**关键需求**：
- 节点级指标：CPU 利用率、内存水位、沙箱密度、网络吞吐
- 沙箱级指标：启动延迟、运行时内存、错误率
- 关键操作的分布式链路追踪（create → 快照恢复 → envd 就绪）
- 结构化日志可被聚合检索（ELK / Loki）
- 告警可接入 PagerDuty / Slack 等通知渠道

---

#### UC-G4：节点扩缩容与维护

> 随业务增长增加计算节点，或在节点维护前将沙箱安全迁走。

**关键需求**：
- 新节点加入后自动参与负载调度，无需手工配置
- 维护节点时将沙箱迁移到其他节点，对用户无感知
- 节点排空期间不接受新沙箱，但已有沙箱继续服务
- 迁移后恢复延迟在可接受范围内（< 30s）

---

### UC-H：安全与合规

#### UC-H1：出站网络管控

> 平台需要控制沙箱内代码可以访问的外部网络目标，防止数据外泄、C2 回连或恶意爬取。

**关键需求**：
- 可按域名/IP 配置允许或拒绝出站连接
- 策略在沙箱运行期间可动态更新
- 可将所有出站流量导向企业自有代理（BYOP）
- 违规出站尝试被记录（审计日志）

---

#### UC-H2：安全代码执行（不受信任代码）

> 平台执行的是来自用户（或 AI 生成）的不可信代码，必须防止代码逃逸到宿主或其他沙箱。

**关键需求**：
- 强隔离边界（硬件虚拟化级别，非容器）
- 沙箱间无任何横向通信路径（二层网络隔离）
- 沙箱内进程无法探测宿主或其他沙箱的存在
- 恶意代码尝试逃逸被记录和告警

---

#### UC-H3：合规审计

> 受监管行业（金融、医疗、政务）要求对沙箱内操作进行完整审计记录，且记录不可篡改。

**关键需求**：
- 所有控制面操作（create/pause/kill）有审计日志
- 审计日志持久化，不随节点重启丢失
- 日志包含：操作者、时间戳、操作类型、沙箱 ID、结果
- 日志可导出到外部 SIEM 系统

---

#### UC-H4：数据静态保护

> 用户数据（上传的文件、运行时产生的数据）在存储和快照中必须加密，防止存储介质泄露。

**关键需求**：
- 运行时写层（overlay）静态加密
- 快照文件传输和存储均加密
- 密钥由用户控制，平台运营商不能解密用户数据
- 沙箱销毁后存储介质安全擦除
